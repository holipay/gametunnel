# 避免 PR 积压的改进方案

## 问题根因

1. **分支无保护** — 任何人都可以直接推送到 main，没有 PR 审查流程
2. **分支未及时 rebase** — 17个分支基于旧的共同祖先（75b7d99），与 main 代码结构已完全不同
3. **重复工作** — 多个分支都做了同样的 vendor 清理（argon2→hkdf），但各自独立
4. **无 CI 检查** — 没有自动化测试验证分支是否与 main 兼容

## 改进措施

### 1. 启用 GitHub Branch Protection
- 禁止直接推送到 main
- 要求 PR 审查（至少1人批准）
- 要求 CI 通过才能合并
- 要求分支是最新的（up-to-date with main）

### 2. 建立分支生命周期规范
- 功能分支从 main 创建，不超过2周
- 每周 rebase 一次 main
- 超过30天未合并的分支自动标记为 stale

### 3. PR 流程规范
- 每个 PR 只做一件事（单一职责）
- PR 描述包含：目的、改动、测试
- 合并前解决所有 review comment

### 4. CI/CD 增强
- PR 触发自动测试
- 检查分支是否与 main 冲突
- 代码覆盖率检查

### 5. 定期清理
- 每月审查未合并的分支
- 删除过时或不再需要的分支
- 合并后立即删除远程分支

## 实施步骤

1. 在 GitHub 仓库设置中启用 Branch Protection Rules
2. 创建 `.github/CODEOWNERS` 文件指定审查者
3. 更新 CI 配置添加 PR 检查
4. 创建 GitHub Action 自动标记 stale 分支
5. 在 README 中添加分支管理规范
