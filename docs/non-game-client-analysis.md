# GameTunnel 非游戏客户端场景分析

> 2026-05-10 讨论整理。

## 一、背景

GameTunnel 当前的客户端是游戏（通过 TUN 虚拟网卡将游戏流量导入隧道）。本文档分析如果客户端场景扩展到文件传输、远程桌面等用途，服务端需要怎样的改造。

## 二、当前架构对非游戏场景的瓶颈

### 2.1 传输层：纯 UDP 中继

当前所有数据走 UDP relay：

```
客户端 A → UDP → 服务端 → UDP → 客户端 B
```

服务端 `handleRelay()` 是无状态转发器：

```go
func (s *Server) handleRelay(payload []byte, from *net.UDPAddr) {
    // 找到目标 → 转发，没有可靠性、流控、重传
    encoded := protocol.EncodeChecked(protocol.TypeData, payload)
    for _, addr := range targets {
        s.sendCheckedRaw(encoded, addr)
    }
}
```

| 场景 | 痛点 |
|------|------|
| 文件传输 | UDP 无拥塞控制，大量数据灌入会丢包爆炸；无重传机制 |
| 远程桌面 | 高频小包（鼠标/键盘 + 屏幕差分），丢包导致画面撕裂 |
| 通用 | `WriteToUDP` 发完就不管，没有发送确认 |

### 2.2 速率限制

```go
const rateLimit = 500 // 每客户端每秒 500 包
```

| 场景 | 需求 | 500 包/秒能提供的带宽 |
|------|------|---------------------|
| 游戏 | 每秒几百个小包 | ~700KB/s，够用 |
| 文件传输 | MB/s 级别 | 仅 700KB/s（MTU 1400），远远不够 |
| 远程桌面 1080p 30fps | 1.5-3MB/s | 勉强，但丢包重传会进一步降低有效吞吐 |

### 2.3 MTU 限制

```ini
mtu=1400
```

| 场景 | 影响 |
|------|------|
| 文件传输 | 每包仅 ~1380 字节有效载荷，协议头开销 14 字节，效率 98.6%（尚可） |
| 远程桌面 | 可能需要更小的包降低延迟 |
| 大文件传输 | 更大的 MTU（如 9000 Jumbo Frame）可提高吞吐 |

### 2.4 缺少的关键能力

| 能力 | 当前状态 | 文件传输需要 | 远程桌面需要 |
|------|----------|-------------|-------------|
| 拥塞控制 | ❌ | ✅ 必需 | ✅ 部分需要 |
| 可靠重传（ARQ） | ❌ | ✅ 必需 | ⚠️ 可选 |
| 流量优先级 | ❌ | ⚠️ 可选 | ✅ 必需（延迟敏感） |
| 多路复用 | ❌ | ⚠️ 可选 | ✅ 区分输入/输出流 |
| 断点续传 | ❌ | ✅ 大文件需要 | ❌ |
| 流控/反压 | ❌ | ✅ 接收方处理不过来时需要 | ⚠️ 可选 |
| 加密 | ❌ | ✅ 文件数据敏感 | ✅ 屏幕内容敏感 |

## 三、服务端改造方案

### 方案 A：参数调优（最小改动）

只改配置，不改架构。

```go
// 放宽速率限制
const rateLimit = 5000  // 10x

// 允许更大 MTU
-mtu 65535  // 支持巨型帧

// 放宽子网限制
-subnet 10.10.0.0/16  // 支持更多客户端
-max 100
```

| 指标 | 改动前 | 改动后 |
|------|--------|--------|
| 最大吞吐 | ~700KB/s | ~7MB/s |
| 最大客户端数 | 10 | 100 |
| 子网大小 | /24（254 IP） | /16（65534 IP） |

**效果**：文件传输能跑起来，但没有可靠性保障。传小文件（<100MB）可以凑合用。

**改动量**：3-5 个文件，0.5 天。

### 方案 B：叠加可靠性层（中等改动）

在 UDP 隧道之上实现可靠性。

```
┌──────────────────────────────────────┐
│  应用层（文件传输/远程桌面）            │
├──────────────────────────────────────┤
│  可靠性层（ARQ / 拥塞控制 / 流控）     │  ← 新增
├──────────────────────────────────────┤
│  GameTunnel UDP 隧道（现有）           │
└──────────────────────────────────────┘
```

**新增模块**：

| 模块 | 说明 | 复杂度 |
|------|------|--------|
| 流分类 | 新增 `flowID` 字段，区分文件/屏幕/键盘流量 | 低 |
| 优先级队列 | 远程桌面 > 文件传输，延迟敏感流量优先转发 | 中 |
| 拥塞信号 | 检测丢包率，通知发送方减速 | 中 |
| 差异化速率限制 | 远程桌面限 1000 包/秒，文件传输限 10000 包/秒 | 低 |
| 缓冲区扩大 | 服务端缓冲从 `maxPlayers * 64` 扩大到 `maxPlayers * 1024` | 低 |

**改动量**：10-15 个文件，2-3 周。

### 方案 C：智能代理（大改动）

服务端变成智能代理，支持多流复用、加密、持久化。

```
┌──────────┐     ┌──────────────────────┐     ┌──────────┐
│ 客户端 A  │────│  GameTunnel 服务端     │────│ 客户端 B  │
│          │◄───│  - 可靠传输层          │◄───│          │
│          │    │  - 流控/拥塞控制       │    │          │
│          │    │  - 流量优先级          │    │          │
│          │    │  - 流量加密            │    │          │
│          │    │  - 断点续传缓存        │    │          │
└──────────┘    └──────────────────────┘     └──────────┘
```

**新增组件**：

| 组件 | 说明 | 参考实现 |
|------|------|---------|
| 传输层抽象 | 在 UDP 之上实现可靠传输 | QUIC（`quic-go`） |
| 流控 | 滑动窗口 + ACK | TCP Reno/CUBIC |
| 加密 | 数据面加密 | Noise Protocol / TLS 1.3 |
| 持久化 | 断点续传需要服务端暂存 | 文件分块 + 偏移记录 |

**改动量**：20+ 文件，1-2 月。

## 四、方案对比

| 维度 | 方案 A（调参） | 方案 B（可靠性层） | 方案 C（智能代理） |
|------|---------------|-------------------|-------------------|
| 改动量 | 0.5 天 | 2-3 周 | 1-2 月 |
| 文件传输吞吐 | ~7MB/s | ~10-50MB/s | ~50-100MB/s |
| 可靠性 | ❌ 无 | ✅ ARQ + 拥塞控制 | ✅ 完整可靠性栈 |
| 远程桌面体验 | 差 | 良好 | 优秀 |
| 代码复杂度增加 | 极小 | 中等 | 大 |
| 维护成本 | 极小 | 中等 | 高 |

## 五、结论与建议

### 5.1 按场景选择

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 偶尔传小文件 | A（调参） | 够用，不值得大改 |
| 经常传大文件 | 不建议用 GameTunnel | Tailscale + scp/rsync 更成熟 |
| 远程桌面 | B（可靠性层） | 延迟敏感，需要优先级队列 |
| 多人协作（文件+桌面+语音） | C（智能代理） | 需要多流复用和差异化 QoS |

### 5.2 战略建议

GameTunnel 的核心价值是**轻量 + 零门槛 + 游戏广播转发**。

文件传输和远程桌面的需求与游戏场景差异较大：

| 维度 | 游戏 | 文件传输 | 远程桌面 |
|------|------|----------|----------|
| 传输协议 | UDP 为主 | TCP 为主 | TCP/UDP 混合 |
| 可靠性 | 容忍丢包 | 不容忍 | 部分容忍 |
| 延迟 | 极敏感 | 不敏感 | 敏感 |
| 带宽 | 低（KB/s） | 高（MB/s） | 中（MB/s） |
| 广播 | 必需 | 不需要 | 不需要 |

**建议**：如果要支持非游戏场景，优先做方案 A（调参）+ 方案 B（可靠性层），不建议做方案 C（智能代理）—— 那个方向应该直接用 QUIC 库或 Tailscale。

GameTunnel 的差异化优势不在「全能」，而在「极简」。
