# GameTunnel 产品差异化发展策略

> 2026-05-10 讨论整理。

## 一、竞争格局

### 1.1 竞品对比矩阵

| 维度 | ZeroTier | Tailscale | WireGuard | GameTunnel（现状） |
|------|----------|-----------|-----------|-------------------|
| 定位 | 通用虚拟 L2 交换机 | 通用 mesh VPN | 纯加密隧道 | 局域网游戏隧道 |
| 架构 | Planet/Moon 层级，依赖控制器 | 协调服务器 + WireGuard mesh | 点对点加密 | 纯自包含，无外部依赖 |
| 协议 | 自有协议（VL1/VL2） | WireGuard + MagicSock | WireGuard | 自有 UDP 协议 |
| TUN 层级 | L2（虚拟交换机） | L3（虚拟路由） | L3（点对点隧道） | L3（虚拟路由） |
| 广播支持 | 支持（L2 天然） | ❌ 不支持 | ❌ 不支持 | ✅ 内置优化 |
| 外部依赖 | daemon + 控制器 | daemon + 账号系统 | 内核模块 | **零依赖，单二进制** |
| 注册要求 | 需要账号 | 需要账号 | 无（手动配置密钥） | **无** |
| 加密 | ✅ | ✅（WireGuard 级） | ✅ | ❌ |
| 代码量 | ~10 万行 C++ | ~5 万行 Go | ~5 万行 C | **~3000 行 Go** |
| 部署复杂度 | 高（注册 + 安装 + 加入网络） | 中（安装 + 登录） | 高（密钥交换 + 路由配置） | **低（下载 + 改 config + 运行）** |

### 1.2 GameTunnel 的核心优势

**优势 1：零摩擦部署**

| 步骤 | ZeroTier | Tailscale | GameTunnel |
|------|----------|-----------|-----------|
| 1 | 注册账号 | 注册账号 | 下载二进制 |
| 2 | 创建网络 | 安装客户端 | 编辑 config.ini（4 个字段） |
| 3 | 安装 daemon | OAuth 登录 | 双击运行 |
| 4 | 加入网络 | 等待协调 | — |
| 5 | 等待控制器分配 | — | — |

**用户心智**：「我只想和朋友打局游戏，不想注册任何东西。」

**优势 2：广播转发是硬需求**

局域网游戏发现依赖 UDP 广播（`255.255.255.255:6112` 等）。覆盖情况：

| 工具 | UDP 广播 | 子网广播 | mDNS 组播 |
|------|----------|----------|-----------|
| ZeroTier | L2 模式支持（配置复杂） | 同上 | 同上 |
| Tailscale | ❌ | ❌ | ❌ |
| WireGuard | ❌ | ❌ | ❌ |
| GameTunnel | ✅ 零配置 | ✅ 零配置 | ✅ 零配置 |

这是经典游戏（星际争霸、红警、帝国时代、暗黑 2）的硬需求。

**优势 3：可审计、可魔改**

3000 行 Go vs ZeroTier 的 10 万行 C++：
- 用户能读完全部代码
- `make server` 一条命令编译
- 没有外部依赖链（Tailscale 依赖 WireGuard 内核模块）

## 二、竞品的盲区

### 2.1 ZeroTier 的弱点

| 弱点 | 说明 |
|------|------|
| 控制器依赖 | 默认依赖 ZeroTier 公司的控制器，自建 Planet 需要额外部署 |
| L2 开销 | 虚拟交换机模式对游戏场景过重，广播流量全量转发 |
| 延迟 | 控制器参与路径建立，首次连接延迟高 |
| 资源占用 | daemon 常驻，内存占用 >50MB |

### 2.2 Tailscale 的弱点

| 弱点 | 说明 |
|------|------|
| 账号绑定 | 必须有 Tailscale 账号（Google/GitHub/Microsoft） |
| 协调服务器 | 依赖 Tailscale 的协调服务器（Headscale 可自建但复杂） |
| 无广播 | 不支持 L2 广播，局域网游戏发现完全失效 |
| 免费版限制 | 100 设备、3 用户 |
| 内核依赖 | WireGuard 需要内核支持（旧系统/嵌入式设备可能没有） |

### 2.3 通用 VPN 工具的共性弱点

| 弱点 | 说明 |
|------|------|
| 不关心应用层 | 不识别游戏流量，不做优先级调度 |
| 不做游戏发现 | 只提供网络层连通，不管上层协议 |
| 配置复杂 | 需要理解子网、路由、DNS 等概念 |

## 三、差异化发展路径

### 路径 1：做「游戏专用隧道」的极致

不和通用 VPN 竞争，专注做游戏场景的最好体验。

| 特性 | 说明 | 竞品做不到的原因 |
|------|------|-----------------|
| 游戏协议识别 | 自动识别游戏流量（端口/协议特征），优先转发 | 通用 VPN 不关心应用层 |
| 广播智能抑制 | 只转发游戏需要的广播包，过滤无关广播 | ZeroTier 全量转发 |
| 延迟优化 | 内核 bypass（`SO_TXTIME`）、UDP GRO/GSO | Tailscale WireGuard 路径固定 |
| 游戏兼容性数据库 | 维护「哪些游戏需要什么配置」的知识库 | 通用工具不管这个 |
| 自动端口映射 | 检测游戏端口，自动配置防火墙规则 | 用户手动处理 |
| P2P 优先 | 打洞成功后完全直连，服务端不参与数据面 | ZeroTier 始终经过控制器 |

**产品形态**：保持极简风格，一个二进制搞定一切。

### 路径 2：做「极客自建基础设施」

针对不信任第三方、喜欢自己掌控的用户群。

| 特性 | 说明 |
|------|------|
| Docker 一键部署 | 在现有 `install-server.sh` 基础上加 Docker/容器化 |
| 多房间/多租户 | 一个服务端实例支持多个独立房间 |
| Web 管理面板 | 状态页升级为管理面板（踢人、查看流量、配置房间） |
| REST API | 管理房间、查看状态，方便集成到其他系统 |
| 自定义认证 | 支持外部认证源（OAuth、LDAP），适合小团队/社区 |
| 流量统计 | 按用户/房间统计流量，方便计费或监控 |

**对标**：自建版 Tailscale Control Server（Headscale），但更轻量、更专注。

### 路径 3：做「轻量级虚拟局域网平台」

在游戏隧道基础上扩展为通用的轻量虚拟局域网工具。

```
┌─────────────────────────────────────────┐
│         GameTunnel Platform             │
├─────────────────────────────────────────┤
│  🎮 游戏隧道（现有核心）                  │
│  📁 局域网文件共享（SMB/NFS）             │
│  🖥️ 局域网远程桌面（RDP/VNC）            │
│  🎤 局域网语音（Mumble/TS3）             │
│  📺 局域网串流（Moonlight）              │
└─────────────────────────────────────────┘
```

**风险**：与 ZeroTier/Tailscale 正面竞争，且它们有 100 倍的资源。

### 路径 4：做「嵌入式/边缘场景的网络隧道」

这是竞品的盲区。

| 场景 | 说明 | 为什么不用 Tailscale |
|------|------|---------------------|
| 路由器固件 | OpenWrt 插件，路由器级别隧道 | Tailscale 太重，吃内存 |
| IoT/嵌入式 | 树莓派、开发板之间的隧道 | WireGuard 需要内核支持 |
| 容器网络 | Docker 容器之间的轻量互通 | 不需要完整 overlay 网络 |
| 游戏主机 | PS/Xbox/Switch 的 LAN 联机 | 无法安装 Tailscale |
| 离线/内网 | 无互联网环境下的局域网扩展 | Tailscale 依赖公网协调服务器 |

**GameTunnel 的独特优势**：
- 纯用户态，不需要内核模块
- 单二进制，资源占用极小（<10MB 内存）
- **可以完全离线运行**（不依赖任何外部服务）

## 四、文件传输与远程桌面场景分析

### 4.1 当前架构对非游戏场景的瓶颈

| 瓶颈 | 游戏场景 | 文件传输 | 远程桌面 |
|------|----------|----------|----------|
| 传输协议 | UDP 为主，容忍丢包 | 需要可靠性 | 需要低延迟 + 部分可靠性 |
| 速率限制 | 500 包/秒够用 | 500 包/秒 = 仅 700KB/s | 勉强够用 |
| MTU 1400 | 合适 | 偏小 | 合适 |
| 拥塞控制 | 不需要 | 必需 | 部分需要 |
| 广播转发 | 必需 | 不需要 | 不需要 |

### 4.2 如果要支持非游戏场景

**方案 A：参数调优（最小改动）**

```go
const rateLimit = 5000  // 放宽 10 倍
// 支持更大 MTU
// 允许更大子网（/16）
```

效果：文件传输 ~7MB/s，但没有可靠性保障。

**方案 B：叠加可靠性层（中等改动）**

在 UDP 隧道之上实现 ARQ + 拥塞控制 + 流量优先级。

新增模块：
- 流分类（区分文件传输 / 远程桌面 / 游戏流量）
- 优先级队列（延迟敏感流量优先）
- 拥塞控制（检测丢包率，通知发送方减速）

改动量：2-3 周。

**结论**：如果主要需求是文件传输或远程桌面，建议直接使用 Tailscale + scp/rsync 或专业远程桌面工具（Parsec/ToDesk）。GameTunnel 应该专注做好游戏场景。

## 五、推荐策略

### 5.1 核心定位

> 「最轻量的自建局域网隧道工具」

不追求功能多，追求**每个功能都做到极致简单**。

### 5.2 路线图

**短期（1-3 月）：巩固游戏场景**

| 优先级 | 特性 | 说明 |
|--------|------|------|
| P0 | 加密（Noise/ChaCha20） | 解决「数据明文」的硬伤 |
| P0 | IPv6 传输层 | 方案 A，改 `"udp4"` → `"udp"` |
| P1 | Web 管理增强 | 房间管理、踢人功能 |
| P1 | 自动重连优化 | 更智能的重连策略 |

**中期（3-6 月）：扩展自建场景**

| 优先级 | 特性 | 说明 |
|--------|------|------|
| P1 | Docker 一键部署 | 降低服务端部署门槛 |
| P1 | 多房间支持 | 一个服务端实例管理多个房间 |
| P2 | REST API | 便于集成和自动化 |
| P2 | WebRTC data channel | 浏览器也能连接 |

**长期：观察 PMF**

- 80% 用户是游戏 → 走路径 1（游戏专用隧道）
- 大量非游戏用途 → 走路径 3（轻量虚拟局域网平台）
- 嵌入式需求多 → 走路径 4（边缘场景隧道）

### 5.3 一句话总结

> ZeroTier 是瑞士军刀，Tailscale 是手术刀，GameTunnel 应该是「开罐器」——只做一件事，但做到没有人比它更简单。

差异化的核心不是功能多，而是**在特定场景下，用户的摩擦成本趋近于零**。
