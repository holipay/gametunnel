# GameTunnel 改进路线图

> 2026-05-19，基于 v1.0.1 代码库全面审查后整理。

## 一、可靠性（直接影响用户体验）

### 1.1 服务端房间持久化

**现状**：服务端所有状态（客户端表、IP bitmap、认证信息）仅存于内存。重启后所有客户端断连、虚拟 IP 丢失、游戏局中断。

**改进方案**：

- 保存已分配的 IP bitmap + 客户端信息到磁盘（JSON 或 bolt.db）
- 重启后恢复状态，客户端重连时能拿到相同的虚拟 IP
- 对星际争霸等需要固定 IP 的游戏场景很关键

**优先级**：🔴 高。这是管理员重启服务时最直接的痛点。

### 1.2 协议版本协商

**现状**：线上格式没有版本号字段。未来如果改协议（比如加 IPv6 虚拟子网），新旧版本客户端混连会直接解析失败、静默丢包。

**改进方案**：

- 在握手阶段（Register / AssignIP）交换协议版本号
- 版本不匹配时给用户明确提示，而不是"连接超时"

**优先级**：🟡 中。当前协议稳定，但为未来扩展扫清障碍。

### 1.3 拥塞控制 / QoS

**现状**：无任何流量整形。如果某个玩家传大文件，UDP 包会挤满发送队列，游戏小包被延迟或丢弃——表现就是所有人同时卡顿。

**改进方案**：

- 按消息类型优先级排队：控制包（KeepAlive/Ping/Pong） > 游戏数据包 > 大文件
- 或者简单限速：单客户端上行带宽上限（如 10 Mbps）
- 发送队列已有 `ctrlCh`（高优先级）和 `sendCh`（普通）的分离，可以在此基础上扩展

**优先级**：🟡 中。对纯游戏场景影响不大（游戏包通常很小），但对"游戏+传文件"混合场景有明显帮助。

### 1.4 TUN 批量读写

**现状**：TUN 读写每次处理一个包（batch=1）。WireGuard TUN 驱动支持批量接口（`Read`/`Write` 多个包），但当前未使用。

**改进方案**：

- 参考 `docs/tun-batch-api-migration.md` 中的方案
- 批量读取减少系统调用次数，高吞吐场景下显著降低 CPU 占用

**优先级**：🟢 低。对游戏小包（几十到几百字节）收益有限，大文件传输场景更明显。

## 二、安全性

### 2.1 前向保密（Forward Secrecy）

**现状**：所有加密都基于房间密码派生的静态密钥（HKDF-SHA256）。如果密码泄露，攻击者可以解密**所有历史流量**。

**改进方案**：

- 握手阶段用 X25519 做 ECDH 密钥交换，每会话生成临时密钥对
- 房间密码只用于认证（HMAC challenge-response），不直接派生加密密钥
- 即使密码泄露，过去的会话无法解密
- 参考 WireGuard 的 Noise_IKpsk2 握手模式

**优先级**：🟡 中。当前密码保护已足够对抗被动嗅探，但主动攻击者（如中间人截获密码）可以解密历史流量。

### 2.2 客户端身份 Session Token

**现状**：认证通过后，客户端只靠 UDP 源地址标识。攻击者如果能伪造源地址（局域网内相对容易），可以注入数据包。

**改进方案**：

- 认证成功后，服务端分配随机 session token（16 字节）
- 客户端每个数据包携带 token（放在现有协议头中）
- 服务端校验 token 与客户端地址的绑定关系
- AEAD 加密已有认证，但仅在有密码时启用；无密码房间需要 token 保护

**优先级**：🟢 低。需要密码的房间已有 AEAD 保护，无密码房间本身就不防攻击。

### 2.3 密码强度校验

**现状**：房间密码无任何强度要求，可以设为 "1" 或空字符串。

**改进方案**：

- 可选：密码长度 ≥ 4，或提示用户使用强密码
- 不做强制限制（保持"零配置"理念），但在 UI 上给出安全提示

**优先级**：🟢 低。

## 三、平台扩展

### 3.1 macOS 客户端

**现状**：macOS 有原生 utun 接口（`/dev/utunX`），实现成本低。很多独立游戏玩家用 Mac。

**技术要点**：

- 使用 `golang.zx2c4.com/wireguard/tun` 的 macOS 实现（已有 darwin 支持）
- TUN 配置：`ifconfig utunX inet6 <ip> <server_ip> netmask 255.255.255.0`
- 路由：`route add -net 10.10.0.0/24 <server_ip>`
- 防火墙：macOS 默认无入站阻断，但需要处理 `pfctl` 规则
- 需要 root 权限或 `setuid` 二进制

**优先级**：🟡 中。扩展用户覆盖面。

### 3.2 Linux GUI 客户端

**现状**：Linux 只能跑服务端。如果能在 Linux 桌面端也作为客户端使用，覆盖开发者群体。

**技术要点**：

- TUN 实现已有（`golang.zx2c4.com/wireguard/tun` 支持 Linux）
- GUI 可用 Fyne（Go 原生跨平台 GUI）或 GTK
- 或者做一个简单的 Web UI（本地 HTTP 服务器 + 浏览器）

**优先级**：🟢 低。Linux 用户通常能接受纯命令行操作。

### 3.3 服务端 Windows 支持

**现状**：服务端只支持 Linux。部分用户可能有 Windows 服务器。

**技术要点**：

- 已有 `socket_windows.go`（socket 缓冲设置）
- TUN 配置代码仅客户端使用，服务端不依赖 TUN
- 主要工作：Windows 服务注册（`sc create`）、防火墙规则自动配置

**优先级**：🟢 低。绝大多数 VPS 是 Linux。

## 四、运维可观测性

### 4.1 时序指标（Time-Series Metrics）

**现状**：状态页只有生命周期累计计数器。无法回答"过去 5 分钟有多少人掉线"、"延迟趋势如何"。

**改进方案**：

- 内存中保留最近 1 小时的滑动窗口数据（环形缓冲区）
- 每分钟采样：在线人数、平均 RTT、丢包率、中转 vs P2P 比例
- 状态页 `/api/metrics` 返回 JSON 时序数据
- HTML 页面展示折线图（内嵌轻量 JS 图表库，如 Chart.js CDN 或纯 SVG）

**优先级**：🟡 中。对管理员排查问题很有帮助。

### 4.2 告警通知

**现状**：服务端异常时（如玩家全部掉线、内存泄漏）无法通知管理员。

**改进方案**：

- 支持 webhook 回调：Discord / Telegram / 企业微信
- 触发条件：在线人数突降（如 5 分钟内从 10 降到 0）、连续认证失败、内存占用超阈值
- 或者简单写日志文件，配合外部监控（Prometheus + Grafana）

**优先级**：🟢 低。当前可通过外部脚本监控 `/api/status` 实现。

### 4.3 结构化日志

**现状**：日志为纯文本 `log.Printf`，难以自动化分析。

**改进方案**：

- 使用 `slog`（Go 1.21+ 标准库）输出 JSON 格式日志
- 字段：timestamp, level, event, username, virtualIP, publicAddr, rtt, error
- 支持日志级别控制（`-loglevel debug/info/warn/error`）

**优先级**：🟢 低。

## 五、用户体验

### 5.1 连接质量可视化

**现状**：客户端只有日志能看到连接状态。玩家不知道自己是 P2P 还是中转、延迟多少。

**改进方案**：

- 系统托盘 tooltip 显示：延迟、丢包率、P2P/中转状态
- 托盘图标颜色变化：绿色（P2P 直连）、黄色（中转）、红色（断开）
- 或者快捷键弹出简洁的状态面板（如 Ctrl+Shift+G）

**优先级**：🟡 中。让玩家直观看到连接质量，减少"为什么这么卡"的困惑。

### 5.2 自动更新检查

**现状**：需要手动下载新版。

**改进方案**：

- 启动时检查 GitHub releases 最新版本（`GET https://api.github.com/repos/holipay/gametunnel/releases/latest`）
- 托盘图标显示更新提示气泡
- 不做自动下载（安全考虑），只提示用户去下载

**优先级**：🟢 低。

### 5.3 多房间支持

**现状**：一个服务端进程只支持一个房间（一个 `roomID`）。管理员想给不同朋友群开不同房间，需要跑多个进程。

**改进方案**：

- 服务端支持多房间，每个房间独立的 IP bitmap、密码、玩家列表
- 客户端 config.ini 中指定 room 即可加入对应房间
- 不同房间之间完全隔离（虚拟 IP 可以重复）

**优先级**：🟡 中。提升单服务器的利用率。

### 5.4 客户端 GUI

**现状**：Windows 客户端只有系统托盘 + config.ini。`docs/gui-client-design.md` 中有设计方案但未实现。

**改进方案**：

- 参考 `docs/gui-client-design.md` 中的 Win32 GUI 方案
- 简洁的设置窗口：服务器地址、玩家名、房间、密码
- 连接状态面板：延迟、丢包、对端列表

**优先级**：🟡 中。降低新用户的使用门槛。
