# GameTunnel 适用范围与应用场景

> 2026-05-10 讨论整理。

## 一、设计定位

GameTunnel 是一个轻量级虚拟局域网工具，核心思路是创建 TUN 虚拟网卡，让不同地区的玩家处于同一个 `10.10.0.0/24` 子网，本质上是造了一个"虚拟局域网"。

与 ZeroTier、Tailscale 等方案相比，GameTunnel 的特点是：
- **零注册**：不需要账号，不需要登录
- **零依赖**：单个 exe + wintun.dll，不需要安装驱动管理器
- **零配置**：config.ini 四个字段，双击即用
- **专为游戏优化**：内置 UDP 广播转发，覆盖绝大多数局域网游戏发现协议

## 二、技术覆盖能力

### 2.1 协议支持

| 层级 | 支持情况 | 说明 |
|------|----------|------|
| IP 层 | ✅ 全部 | TUN 工作在 IP 层，TCP/UDP/ICMP 均可转发 |
| IPv4 | ✅ | 当前唯一支持 |
| IPv6 | ❌ | 未实现 |

### 2.2 发现机制覆盖

| 发现方式 | 覆盖 | 典型游戏/应用 |
|----------|------|---------------|
| UDP 广播（255.255.255.255） | ✅ | 星际争霸、红警、帝国时代、暗黑2 |
| 子网广播（x.x.x.255） | ✅ | 大多数老 RTS、局域网游戏 |
| mDNS（224.0.0.251） | ✅ | 部分 macOS/iOS 游戏、星际1 LAN 发现 |
| TCP 直连 IP | ✅ | 只要知道对端虚拟 IP |
| SSDP/UPnP（239.255.255.250） | ✅ | DLNA 设备发现 |
| Steam 好友/大厅 | ❌ | 走 Steam 自己的网络，不走局域网 |
| 第三方 P2P 穿透 | ❌ | 游戏自有的穿透方案不受控制 |

### 2.3 路由架构

```
所有玩家获得 10.10.0.x 虚拟 IP
  │
  ├─ 默认路由 0.0.0.0/0 → TUN（metric=1）
  │   └─ 所有游戏流量自动进隧道
  │
  ├─ 广播路由 255.255.255.255 → TUN
  │   └─ UDP 广播发现走隧道
  │
  ├─ mDNS 路由 224.0.0.251 → TUN
  │   └─ 组播发现走隧道
  │
  └─ 服务器排除路由 → 物理网卡（metric=1）
      └─ 隧道 UDP 流量不回环
```

### 2.4 连接模式

| 模式 | 条件 | 延迟 |
|------|------|------|
| P2P 直连 | 双方至少一方在锥形 NAT 后 | 取决于双方到彼此的延迟 |
| 服务器中转 | 打洞失败或对称 NAT | 取决于到服务器的延迟（国内 VPS 通常 20-50ms） |

## 三、适用场景

### 3.1 经典局域网游戏（核心场景）

这是 GameTunnel 的主要设计目标。大量经典游戏的联机功能基于局域网广播发现：

- **RTS**：星际争霸、魔兽争霸3、红色警戒、帝国时代2、命令与征服
- **ARPG**：暗黑破坏神2、火炬之光
- **赛车**：极品飞车部分版本
- **FPS**：CS 1.6 局域网模式、求生之路

这些游戏通常：
1. 启动后扫描局域网广播
2. 发现同网段的房间列表
3. 直接 UDP/TCP 连接

GameTunnel 的广播转发机制完美覆盖此流程。

### 3.2 远程办公与内网访问

跳出游戏场景，虚拟局域网在以下场景同样有用：

| 场景 | 说明 |
|------|------|
| 访问公司内网资源 | 文件共享（SMB）、内网 Wiki、ERP、OA 系统 |
| 远程桌面 | Windows RDP、VNC 走局域网发现，不用记 IP |
| 打印机/扫描仪 | 很多办公设备只在局域网内可发现 |

### 3.3 开发与测试

| 场景 | 说明 |
|------|------|
| 多机联调 | 开发机在不同网络（家/公司/咖啡厅），需要模拟同一子网 |
| 容器/微服务测试 | 跨物理机的容器需要像在同一网络里通信 |
| 数据库主从复制 | MySQL、Redis 集群节点间通信 |

### 3.4 NAS 与家庭服务

| 场景 | 说明 |
|------|------|
| 远程访问 NAS | SMB/NFS/AFP 协议设计给局域网用，走公网需额外配置 |
| DLNA / AirPlay 投屏 | 发现协议依赖局域网广播 |
| 智能家居 | Home Assistant 发现设备走 mDNS / SSDP |

### 3.5 影音制作

| 场景 | 说明 |
|------|------|
| NDI 视频传输 | 多机直播/导播，NDI 协议依赖局域网发现和组播 |
| DAW 联机 | 多机跑 Cubase/Nuendo 做分布式音频渲染 |

### 3.6 安全与隐私

| 场景 | 说明 |
|------|------|
| 不暴露端口到公网 | 服务只在虚拟局域网内可达，比开公网端口 + 防火墙更安全 |
| 配合 WireGuard 加密 | GameTunnel 明文传输，加密需求可叠加 WireGuard |
| 绕过 CGNAT | 家里没公网 IP，但又需要从外面访问内网设备 |

## 四、局限性

### 4.1 不适用的场景

| 场景 | 原因 |
|------|------|
| Steam/Epic 等平台联机 | 走平台自己的网络栈，不走局域网 |
| 无"局域网模式"的游戏 | 只能通过平台匹配，无法隧道 |
| IPv6 专用应用 | 当前不支持 IPv6 |
| 延迟敏感的竞技 FPS | 中转模式 20-50ms 可能不够 |

### 4.2 潜在风险

| 风险 | 说明 |
|------|------|
| 反作弊检测 | 某些反作弊系统可能检测到虚拟网卡并拒绝运行（概率不大） |
| 网卡绑定 | 少数游戏会枚举物理网卡 IP 而不是走默认路由 |
| 明文传输 | 游戏数据不加密，需要加密请配合 WireGuard |

## 五、与其他方案对比

| 方案 | 特点 | 适合场景 |
|------|------|----------|
| **GameTunnel** | 轻量、无依赖、广播转发、零配置 | 游戏、临时组网、快速搭建 |
| ZeroTier | 功能全、有控制面板、需要注册 | 长期办公、固定团队 |
| Tailscale | WireGuard 内核、Mesh 网络 | 安全要求高的企业场景 |
| WireGuard | 纯加密隧道、点对点 | 安全要求高、网络知识强 |
| FRP / ngrok | 端口转发、不需要虚拟网卡 | 只需暴露单个服务 |

**GameTunnel 的差异化优势**：不需要注册账号、不需要安装驱动管理器、不需要理解路由表。对于"几个人临时联个网"的场景，门槛比 ZeroTier 低很多。

## 六、总结

GameTunnel 的核心价值是**轻量 + 零门槛**。

在游戏场景中，它覆盖了绝大多数基于 IP 的局域网游戏，特别是依赖 UDP 广播发现的经典游戏。在非游戏场景中，任何需要"像在同一局域网一样"访问服务的场合都可以使用，但长期、大规模的组网需求建议使用 ZeroTier/Tailscale 等成熟方案。
